A step-by-step guide to our agentic QE framework — how the team turns a user story or Jira issue into Page-Object test suites, runs them, and self-heals failures, with human approval at every gate.

intake→plan→ approve→generate→ run→heal
01 The idea in one screen

Automation that starts from intent, not from code

The old way

Read the story → hand-write locators → hand-write a Page Object → hand-write a spec → debug flakiness → repeat for every language. Hours per feature, brittle by Friday.

◆ The framework way

You supply a requirement. A planner agent explores the live app, proposes an evidence-based plan; you approve; a generator agent writes conventions-perfect code and runs it; a healer agent root-causes any failure.

You stay in control. Agents do the typing and the exploring. Humans decide what to test (approve the plan) and whether a fix is right (approve the heal). Nothing ships past you.

03 Why the gates matter

Two moments where a human must look

gate 1 Approve the plan

The planner classifies every scenario as VERIFIED (it saw the behavior), ASSUMPTION_NEEDS_REVIEW, or REJECTED_UNSUPPORTED. You confirm scope and the proposed locators before a single line is generated — cheap to change here, expensive later.

gate 2 Approve the heal

When a test fails, the healer decides: application bug (report it, don't touch the test) vs automation issue (propose the minimal fix). You approve before the change lands — assertions are never weakened to force a pass.

These two gates are the difference between "AI wrote some tests" and "a QE process you can trust in CI."

step 1 Intake — bring a requirement

Normalize the story into testable criteria

The story-intake skill turns any source — pasted text, a stories/*.md file, or a Jira issue — into one consistent shape: atomic, testable acceptance criteria the planner can consume.

stories/example-login.md
As a registered user
I want to sign in with email + password
So that I reach my workspace.

Acceptance criteria
- valid credentials → land on workspace
- invalid credentials → inline error
- empty email → submission blocked
- UI available in English + Arabic
  • Every criterion is atomic. Compound ones get split so each maps to one test.
  • Missing criteria become open questions — the agent flags them, it never invents requirements.
  • The source id is preserved (e.g. SCRUM-142) so traceability tags survive into the test titles.
step 2 Plan — explore the live app

Evidence, not assumptions

The pw-test-planner opens the real app with the Playwright MCP, inspects the DOM, and captures the exact locators and behaviors. It writes a plan to specs-plans/<slug>.plan.md — and writes no test code.

  • Real locators, chosen by the priority ladder and proven against the live DOM.
  • Real behaviors — it confirms what the app actually does vs what the story assumes.
  • Every scenario classified VERIFIED / NEEDS_REVIEW / REJECTED, with the evidence it saw.
  • Open questions for QA surfaced up front.
specs-plans/osool-login.plan.md
[OSOOL-LOGIN-01] valid login → /workspace/projects
  VERIFIED · @smoke @login @p0
[OSOOL-LOGIN-02] invalid creds → #loginMessage
  VERIFIED · @regression @login @p1
# evidence: email #email · pass #password-field
# login btn button.signIn-createBtn  (NOT
# button[type=submit] — matches hidden OTP)
step 3 Gate — review & approve the plan

Your five-minute checklist

  • Scope — are the right scenarios there, and nothing out of scope?
  • Classifications — is anything marked VERIFIED that you doubt? Push back.
  • Locators — do the proposed selectors look stable (ids/testids over positional)?
  • Data & tags — correct priority, correct feature tag, traceable case ids?
  • Open questions — answer them before generation.
Approve by telling Claude to proceed. Only then does pw-test-generator run.

Nothing is generated yet

At this point the repo has a plan file and your comments — zero test code. Changing direction is free.

step 4 Generate — from the approved plan

Conventions-perfect code, written for you

The pw-test-generator produces a full, runnable slice and then executes it — Page Object, spec, both locale files, and test data, each following the POM conventions exactly.

pages/LoginPage.ts POM class tests/ui/login.spec.ts the scenarios locales/en/login.ts strings (fallback) locales/ar/login.ts matching keys test-data/testing/… data + secrets via env
  • POM only — one class per page, mandatory member order.
  • Every action via WebUtil — never page.click directly.
  • Strings from getLocale() — en + ar kept in lockstep.
  • Tags from utils/tags, titles as [caseId] validate that …
step 5 Run — by tag, title, language, device

Execute the suite

bash
npx playwright test                        # everything
npx playwright test tests/ui/login.spec.ts # one file
npx playwright test --grep @smoke          # by tag
LANGUAGE=ar npx playwright test            # run the Arabic pass
npm run test:smoke   ·   npm run typecheck

Tags

@smoke @regression · @p0…@p2 · @login @ui

Localization

LANGUAGE drives both the app UI and the asserted strings — same suite, both languages.

Platform

PLATFORM_TYPE=desktop/mobile/both

step 6 Report — read the results in Allure

Rich, shareable test reporting

bash
npm test                 # writes allure-results/
npm run allure:serve     # generate + open
# or split:
npm run allure:generate
npm run allure:open
  • Pass / fail totals and history at a glance.
  • Environment panel — ENV, LANGUAGE, PLATFORM, BASE_URL captured automatically.
  • Tags mapped — filter by @smoke, @login, priority.
  • On failure — screenshot, video and trace attached to the test.
Allure runs a local web server (the report uses fetch) — open the printed http://localhost URL. Needs Java on the PATH.
step 7 Gate — heal a failure

Root cause first, fix second

Run /qe-heal <spec-or-trace>. The healer reads the error, trace and screenshots, then classifies the failure before proposing anything.

app bug

The application is wrong. The healer reports it with evidence and does not alter the test. You file the bug.

automation issue

The test is wrong — a brittle locator, a missing wait. The healer proposes the minimal fix and waits for your approval to apply it.

Real example from Osool: the error text flashed "Verifying credentials…" before settling → classified as an automation timing issue → fixed with a web-first auto-retrying assertion, not a weaker check.
07 The conventions the agents enforce

The house rules — cheat sheet

POM structure

Elements → constructor → fills → clicks → uploads → selects → validations. One class per page.

WebUtil for every action

Clicks, fills, reads route through WebUtil — it handles loader waits + step logging.

Localization

Strings via getLocale('feature'). Matching en/ar keys; en is the fallback.

Locator priority

data-testid → CSS → semantic XPath → localized XPath

Banned: absolute XPath, deep nth-child.

Tags & titles

Tags from utils/tags/tags.ts — never hardcode. Titles: [caseId] validate that …

Never weaken assertions

A fix that makes a test pass by asserting less is not a fix. Ever.

09 Do & don't

Keep the process honest

✓ Do

  • Read the plan before approving — it's the cheapest place to steer.
  • Let the planner explore the live app so locators are real.
  • Run both languages before calling a feature done.
  • Treat a flaky external env with retries, not weaker asserts.

✕ Don't

  • Skip a gate to "save time" — that's where trust is built.
  • Hand-edit generated locators inline in a spec.
  • Hardcode strings or tags — use getLocale and TAGS.
  • Commit or push unless you're explicitly asked to.
10 Start now

Build your first suite in five moves

  1. Drop a story in stories/ (or have the Jira key ready).
  2. Run /qe-from-story stories/your-story.md.
  3. Review the plan in specs-plans/ and approve.
  4. Let it generate & run; watch the CLI output.
  5. Open npm run allure:serve and share the report.
Stuck on a failure? /qe-heal <spec> — it tells you whether it's your test or the app.
